# 本次性能优化变更说明

## 1. 背景

本次优化主要针对以下页面加载慢的问题：

- `项目`
- `项目 -> 需求详情`
- `需求管理`
- `采集任务管理`
- `数据验收`
- `调度`

此前页面普遍存在两类性能问题：

1. 前端首屏通过多轮接口拼装页面数据，形成明显的 fan-out / N+1 请求。
2. 后端部分列表接口默认返回过重数据，尤其是任务列表接口会为每个任务额外加载 `collectionRows`，放大了单次请求成本。

本次优化目标是把当前平台的取数模式从：

- `前端多轮拼装`
- `列表接口默认带明细`

收敛为：

- `页面级概览接口`
- `轻量列表接口`
- `按需加载明细`

---

## 2. 主要问题定位

### 2.1 前端问题

- `项目` 页先拉项目，再按项目逐个拉需求。
- `采集任务管理` 页先拉项目，再按项目拉需求，再按需求分别拉任务组和采集任务。
- `数据验收` 页自己拼项目、需求、任务组、验收单。
- `调度` 页为了补任务组上下文，会再次走项目 -> 需求 -> 任务组链路。
- `需求详情` 页原先会加载整个项目下的 `requirements + wideTables`，并额外拉全量验收单、全量调度记录。

### 2.2 后端问题

- `RequirementQueryService.listFetchTasks(...)` 默认逐任务加载 `collectionRows`，存在服务端 N+1。
- `task-runtime` 虽然合并了 `task-groups + tasks`，但内部仍复用了重的任务列表装配逻辑。
- `acceptance-tickets`、`schedule-jobs`、`wide-table rows` 缺少足够细的过滤或分页能力，容易被前端全量拉取。

---

## 3. 本次优化策略

### 3.1 总体策略

本次优化按以下原则实施：

- 列表页首屏优先使用页面级概览接口
- 列表接口默认返回轻量摘要数据
- 明细数据通过专门接口按需加载
- 需求详情只围绕当前需求域取数，不再围绕整个项目域取数
- 对跨页面重复使用的基础数据增加前端缓存

### 3.2 实施阶段

本次变更按 5 个阶段执行并已落地：

1. 阶段 1：后端列表接口去明细、补过滤/分页能力
2. 阶段 2：需求详情收紧到当前需求域取数
3. 阶段 3：采集任务管理改为页面级概览接口
4. 阶段 4：项目页与数据验收页改为聚合接口
5. 阶段 5：调度页与通用缓存优化

## 4. 后端变更说明

## 4.1 轻量 runtime / tasks 能力

### 变更内容

- 为任务列表和任务 runtime 增加 `include_collection_rows` 控制参数。
- 当前端传入 `include_collection_rows=false` 时，列表接口不再默认带 `collectionRows`。

### 影响接口

- `GET /api/projects/{projectId}/requirements/{requirementId}/tasks`
- `GET /api/projects/{projectId}/requirements/{requirementId}/task-runtime`
- `GET /api/tasks?project_id=...&requirement_id=...`
- `GET /api/tasks/runtime?project_id=...&requirement_id=...`

### 主要收益

- 避免任务列表接口默认逐任务加载结果行
- 降低 `采集任务管理`、`需求详情`、任务相关页面的单次请求成本

## 5.2 需求详情页

### 优化前

- 调用 `loadRequirementDetailData(projectId, requirementId)` 时会：
  - 拉项目
  - 拉整个项目的全部需求和宽表
  - 拉当前需求的 runtime
  - 拉全量验收单
  - 拉项目下全部宽表行
  - 拉全量调度记录

### 优化后

- 只拉当前需求：
  - 当前项目
  - 当前需求
  - 当前需求的宽表
  - 当前需求的 runtime
  - 当前需求的验收单
  - 当前需求任务组对应的调度记录
- 去掉了项目域全量数据装配

### 补充说明

- 当前需求页仍保留当前需求范围内的 `wideTableRecords / taskGroups / fetchTasks` 首屏加载。
- 原因是 [RequirementDefinitionForm.tsx](/E:/huatai/datafoundry_java/data-foundry-frontend/components/RequirementDefinitionForm.tsx) 本身就依赖这些数据进行展示和交互，不能简单延后。

## 8. 兼容性说明

本次优化遵循“兼容优先”的原则：

- 旧接口没有被删除
- 新能力通过可选参数或新增概览接口提供
- 页面切换为新接口后，旧接口仍可保留给其他页面或后续渐进迁移使用

## 9. 联调建议

建议按以下顺序联调：

1. `项目`
- 确认项目数量、需求数量展示正确

2. `项目 -> 需求详情`
- 确认进入详情时不再有明显的长时间空白
- 确认需求定义页、任务页、验收页展示仍正确

3. `采集任务管理`
- 确认首屏加载时间明显缩短
- 确认展开任务组、刷新任务实例时数据正确

4. `数据验收`
- 确认待验收、已验收列表数量和跳转正确

5. `调度`
- 确认调度记录展示正常
- 确认任务组上下文名称可正确映射

## 10. 验收重点

- `项目` 页首屏不再出现“按项目逐个拉需求”的请求模式
- `需求详情` 不再加载项目级全量需求、全量宽表、全量验收单、全量调度记录
- `采集任务管理` 首屏改为单次概览请求
- 任务列表接口在轻量模式下不再默认带 `collectionRows`
- 调度和验收页不再依赖前端自行拼装全链路数据
