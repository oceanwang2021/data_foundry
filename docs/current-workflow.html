AI 采数平台当前业务流程

围绕“1 Requirement = 1 WideTable”的主模型组织需求定义、Demo 验证、计划落地、持续运行与验收交付，统一覆盖按业务日期增量采集与按快照全量采集两类链路。

业务方 平台配置 调度 / 执行 治理 / 交付
阶段一 · 需求建模
先稳定目标宽表，再讨论如何执行
①Business
提出采数需求
明确业务目标、交付口径、目标主体与需要持续更新还是快照采集。
→
②Config
创建 Demo 需求并继承项目默认信源
注入行业背景、搜索引擎、站点策略、知识库与固定 URL 等默认配置。
→
③Config
定义宽表、采集模式与指标组
固定 `1 Requirement = 1 WideTable`，维护 Schema、Scope、IndicatorGroup、ScheduleRule。
增量：business_date 全量：full_snapshot
→
④Config
生成预览行并确认范围
先物化 WideTableRow，校验业务日期、维度枚举与行绑定键是否符合预期。
阶段二 · Demo 验证
小批量验证采集表达与结果质量
⑤Execute
生成 Demo 计划版本
基于预览结果生成 CollectionBatch、TaskGroup 与 FetchTask，形成当前可执行计划。
→
⑥Execute
按“行 × 指标组”执行采集
FetchTask 下发给 Agent，执行期再按指标键展开 RetrievalTask 与 NarrowIndicatorRow。
→
⑦Govern
回填宽表并检查 Demo 结果
写回指标值、来源链接、摘录与执行记录，观察结果是否满足业务预期。
→
⑧Business
Demo 通过，原地转正式
需求不复制，直接从 Demo 进入 Production，并保留同一条业务链路。
阶段三 · 计划落地与持续运行
锁定结构，按模式持续执行
⑨Config
锁定 Schema 并确认更新方式
Production 阶段锁定表结构，仅允许扩展范围、调整调度、补采与数据更新方式。
→
⑩Execute
按模式生成批次与任务组
增量模式按业务日期生成批次与任务组；全量模式按整表快照生成 `full_table` 任务组。
→
⑪Execute
历史补数 + 未来调度
历史日期走 BackfillRequest，当前与未来走 ScheduleRule；失败任务按 auto-retry 规则重试。
→
⑫Govern
后处理、稽核与状态聚合
统一更新 WideTableRow、TaskGroup、CollectionBatch 状态，为验收与下游交付准备可消费结果。
阶段四 · 验收交付
按单元格追踪问题并闭环修复
⑬Business
查看宽表结果与单指标溯源
按单元格查看来源链接、原文摘录、发布时间和关联任务，定位问题不再依赖整批人工排查。
→
⑭Govern
人工修订或发起重跑
支持单元格修订、单任务重跑、任务组重跑，以及增量模式下的区间补采。
→
⑮Govern
结果回写并形成新一轮版本
修订与重跑结果继续回写宽表，形成新的执行记录、计划版本与可追溯交付资产。
→
⑯Govern
持续交付宽表资产
面向后处理、稽核、验收和下游消费持续提供结构稳定、可追溯、可重跑的数据结果。
未通过时并不推翻整条链路，而是沿着单元格 → 任务 → 任务组 → 补采区间逐级修复。这样既保留了计划与执行痕迹，也避免了整批重算带来的额外成本。
