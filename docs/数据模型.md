# 数据模型

## 1. 总体模型

当前系统以项目（Project）作为业务容器，以需求（Requirement）作为核心聚合根，以宽表（WideTable）作为数据定义与执行边界。当前正式口径下：

- 1 个 Requirement 对应 1 张 WideTable。
- Demo 需求转正式采用原地转换，不再派生新的正式需求记录。
- 系统同时支持两种采集模式：
  - 按业务日期增量采集
  - 按快照全量采集

系统主流程按下面顺序展开，而不是先生成任务、再倒推数据结构：

1. 在项目下定义需求的业务目标、边界与采集策略。
2. 在需求下定义目标宽表。
3. 在宽表中定义表结构（WideTableSchema）、范围（Scope）、指标组（IndicatorGroup）与调度规则（ScheduleRule）。
4. 先生成宽表行（WideTableRow），再生成采集批次（CollectionBatch）、任务组（TaskGroup）与采集任务（FetchTask）。
5. 执行期按指标展开检索任务与窄表载体，执行后回填宽表。

```mermaid
flowchart LR
    P["项目（Project）"] --> R["需求（Requirement）"]
    R --> WT["宽表（WideTable）"]
    WT --> DEF["表结构 / 范围 / 指标组 / 调度规则"]
    DEF --> ROW["宽表行（WideTableRow）"]
    ROW --> CB["采集批次（CollectionBatch）"]
    CB --> TG["任务组（TaskGroup）"]
    TG --> FT["采集任务（FetchTask）"]
    FT --> RET["检索任务 / 窄表载体"]
    RET --> WB["宽表回填"]
```

其中：

- 宽表行是数据初始化与结果回填的锚点。
- 采集批次是“本轮采集覆盖范围”的抽象。
- 任务组是调度单元。
- 采集任务是执行单元。
- 检索任务与窄表载体属于执行产物层。

## 2. 核心实体

| 实体 | 职责 | 关键关系 |
| --- | --- | --- |
| Project | 承接行业、专题或客户场景，提供默认背景与信源配置 | 组织多个 Requirement |
| Requirement | 定义业务目标、业务边界、交付范围与采集策略 | 聚合 1 张 WideTable |
| WideTable | 表示一张具体的数据表，是记录初始化、计划生成、调度和执行的边界 | 拥有 Schema、Scope、IndicatorGroup、ScheduleRule |
| WideTableSchema | 定义宽表列结构 | 描述 ID 列、维度列、指标列、系统列 |
| IndicatorGroup | 定义应当一起采集的一组指标列 | 决定 FetchTask 如何拆分，可附带 Agent 与 Prompt 配置 |
| ScheduleRule | 定义自动触发方式 | 作用于 TaskGroup / CollectionBatch 的生成与执行 |
| WideTableRow | 表示宽表初始化后的单行记录 | 是 FetchTask 回填的目标对象 |
| CollectionBatch | 表示一次采集覆盖范围或快照波次 | 组织 TaskGroup，并承载快照标签与覆盖模式 |
| TaskGroup | 表示某张宽表在某个分区下的一批任务 | 是调度、补采和批量执行的主单位 |
| FetchTask | 实际执行的最小业务单元 | 等于某张宽表中某个行在某个指标组上的一次采集 |
| RetrievalTask | 单指标检索任务 | 作为 FetchTask 的执行产物，记录 query、状态与窄表结果 |
| BackfillRequest | 表示指定业务日期范围内的补采或历史重跑请求 | 为相关业务日期创建或激活 TaskGroup |
| ExecutionRecord | 记录执行次数、状态、触发来源和失败原因 | 作为 FetchTask 的执行日志 |
| ScheduleJob | 记录一次调度触发 | 面向手动触发、定时执行、补采和重试的运行条目 |

## 3. 宽表模式、表结构与范围

### 3.1 宽表模式

宽表是否具有业务日期，不再由文档约定死，而由模型字段决定。当前用两个字段描述：

- `semantic_time_axis`
  - `business_date`
  - `none`
- `collection_coverage_mode`
  - `incremental_by_business_date`
  - `full_snapshot`

二者的组合含义如下：

1. `business_date + incremental_by_business_date`

   表示按业务日期增量采集。其约束为：

   - 必须存在且仅存在 1 个业务日期维度。
   - `scope.business_date` 必须存在，且必须指向该业务日期维度。
   - 宽表行按 `(业务日期, 其他维度组合)` 初始化。
   - 任务组通常按业务日期拆分。

2. `none + full_snapshot`

   表示按快照全量采集。其约束为：

   - 不允许存在业务日期维度。
   - `scope.business_date` 必须为空。
   - 宽表行仅按非日期维度组合初始化。
   - 采集批次与任务组按整表快照组织。

因此，当前数据模型已经同时覆盖“增量模式”和“全量模式”，两者在同一套实体内通过时间语义和覆盖模式加以区分。

### 3.2 表结构

每张宽表至少包含四类列：

1. ID 列
   - 必须为整数型主键列。
2. 维度列
   - 承载主体、地区、时间等维度。
   - 在增量模式下，必须且只能有 1 个维度列被标记为业务日期。
3. 指标列
   - 承载待采集指标。
   - 每个指标列必须定义 `unit`。
4. 系统列
   - 承载行状态、最近任务、更新时间等系统维护信息。

宽表结构使用 `WideTableSchema` 表示，列键必须唯一。

### 3.3 范围

宽表范围使用 `WideTableScope` 表示，包含：

1. `business_date`
   - 仅在增量模式下存在。
   - 支持 `monthly` 与 `yearly`。
   - 支持最新年度按季度展开。
2. `dimensions`
   - 每个非日期维度列都必须提供枚举值列表。

范围的职责是将自然语言描述收敛为可枚举、可初始化的记录集合。

### 3.4 宽表行初始化

宽表行初始化是规则过程，不依赖 AI。当前初始化原则为：

1. 先根据宽表模式展开时间范围：
   - 增量模式展开业务日期。
   - 全量模式不展开业务日期。
2. 再对非日期维度列取笛卡尔积。
3. 为每个组合生成一条宽表行，并分配整数型 `row_id`。

每条宽表行至少包含：

- `row_id`
- `plan_version`
- `row_status`
- `dimension_values`
- `business_date`
- `row_binding_key`
- `indicator_values`
- `system_values`

其中：

- `row_binding_key` 由业务日期与维度值拼装得到，用于稳定识别同一条业务行。
- `indicator_values` 在初始化时通常为空，等待后续回填。

当前行状态包括：

- `initialized`
- `collecting`
- `partial`
- `completed`
- `invalidated`

## 4. 指标组、采集批次、任务组与采集任务

### 4.1 指标组

一条宽表行不一定只对应一个采集任务。如果同一行中的指标列需要使用不同 Agent、Prompt、优先级或信源偏好，就应先在宽表内定义多个指标组（IndicatorGroup）。

当前约束为：

- 每个指标组至少包含一个指标。
- 组内指标键不可重复。
- 组间指标不可重叠。
- 所有指标组必须覆盖宽表全部指标列。

### 4.2 采集批次

CollectionBatch 用于描述一次采集覆盖范围，是当前模型中连接“宽表行”和“任务组”的中间层：

- 对于增量模式，它承载某一波业务日期执行的批次信息。
- 对于全量模式，它承载某次整表快照的执行信息。

CollectionBatch 的关键字段包括：

- `snapshot_at`
- `snapshot_label`
- `coverage_mode`
- `semantic_time_axis`
- `status`
- `is_current`
- `plan_version`
- `triggered_by`

### 4.3 任务组

任务组（TaskGroup）是调度单元，不再固定等价于“某宽表某业务日期”，而是通过分区类型来定义边界。当前主要分区类型包括：

- `business_date`
- `full_table`

这意味着：

- 在增量模式下，任务组通常表示“某张宽表在某个业务日期下的一批任务”。
- 在全量模式下，任务组通常表示“某张宽表在某个快照批次下的一批整表任务”。

TaskGroup 的关键字段包括：

- `source_type`
  - `scheduled`
  - `backfill`
- `status`
  - `pending`
  - `running`
  - `partial`
  - `completed`
  - `invalidated`
- `partition_type`
- `partition_key`
- `partition_label`
- `schedule_rule_id`
- `backfill_request_id`
- `total_tasks`
- `completed_tasks`
- `failed_tasks`
- `triggered_by`

### 4.4 采集任务

采集任务（FetchTask）的展开规则为：

```text
FetchTask = WideTableRow × IndicatorGroup
```

也就是说：

- 系统先确定一组宽表行。
- 再将每条宽表行与适用指标组进行组合。
- 每个组合生成一条采集任务。

每个 FetchTask 只负责自己那组指标列的采集与回填。

### 4.5 检索任务与窄表载体

采集任务发送给执行器时，会将自己负责的指标列进一步展开为单指标载体。若某指标组包含 N 个指标列，则执行期会生成 N 条单指标请求。

这一层在当前模型中的体现包括：

- RetrievalTask：记录单指标 query、状态、置信度等
- NarrowIndicatorRow：记录单指标对应的维度值、指标定义、结果、来源、摘录等

这些对象的定位是：

- 它们属于执行产物层，不是新的主流程实体。
- 它们的存在是为了支撑调试、溯源、展示和后续分析。

## 5. 调度、补采与生命周期

### 5.1 调度规则

调度规则（ScheduleRule）作用于任务组和采集批次，而不是直接作用于采集任务。

在增量模式下，调度规则负责：

- 为当期与未来业务日期创建或激活 TaskGroup
- 触发这些 TaskGroup 下的 FetchTask 执行

在全量模式下，调度规则负责：

- 为某次快照创建 CollectionBatch
- 再为该批次创建 `full_table` 类型的 TaskGroup

### 5.2 补采请求

补采请求（BackfillRequest）用于指定业务日期范围内的历史补采或重跑，其适用前提为：

- 宽表采用 `business_date + incremental_by_business_date` 模式

也就是说：

- 增量模式支持按业务日期范围补采
- 全量快照模式不支持按业务日期补采

补采请求会：

1. 确定目标业务日期范围。
2. 为这些日期创建或激活对应 TaskGroup。
3. 再按 `WideTableRow × IndicatorGroup` 生成 FetchTask。

### 5.3 执行记录与调度记录

当前模型将两类记录分开：

1. ExecutionRecord
   - 绑定到 FetchTask
   - 描述单次执行尝试
   - 记录触发方式、状态、开始结束时间、错误和日志引用

2. ScheduleJob
   - 绑定到 TaskGroup 或单任务触发
   - 描述一次调度层面的运行条目
   - 用于展示手动触发、定时执行、补采和重试的运行情况

### 5.4 计划版本与失效

宽表结构、范围或指标组变化后，原有计划可能失效。当前模型通过 `plan_version` 承载计划版本语义：

- 宽表行带有 `plan_version`
- 任务组带有 `plan_version`
- 采集任务带有 `plan_version`

这样可以保证：

- 新旧计划可以被区分
- 旧任务组可以被标记为失效或归档
- 当前有效计划始终以最新版本为准

## 6. Demo 与 Production 的模型边界

### 6.1 Demo 需求

Demo 需求是主流程的起点，用于稳定以下内容：

- 需求业务目标与边界
- 宽表结构
- 范围定义
- 指标组划分
- Prompt 表达方式

Demo 需求的特征是：

- `phase = demo`
- `schema_locked = false`
- 允许持续调整宽表定义与计划

### 6.2 Production 需求

Production 需求不是一条独立派生模型，而是同一条 Requirement 在阶段上的演进结果。当前主流程采用原地转换：

```text
Requirement: demo -> production
```

转换后的特征是：

- `phase = production`
- `schema_locked = true`
- 正式进入持续执行与交付阶段

模型层面的约束为：

- Production 阶段锁定 Schema
- 在不改 Schema 的前提下，可以继续调整范围、调度规则、数据更新方式和补采请求

### 6.3 数据更新模式

当前 Requirement 上的 `data_update_mode` 与宽表模式保持一致：

- `collection_coverage_mode = incremental_by_business_date` 对应 `data_update_mode = incremental`
- `collection_coverage_mode = full_snapshot` 对应 `data_update_mode = full`

因此，需求层的数据更新语义并不是独立存在的，而是从宽表采集模式派生而来。

## 7. 小结

当前数据模型的核心收敛可以概括为：

1. 主模型收敛为 `Project -> Requirement -> WideTable`，其中 `1 Requirement = 1 WideTable`。
2. 通过 `semantic_time_axis` 与 `collection_coverage_mode`，在同一套模型中统一表达“增量采集”和“全量采集”。
3. 通过 `WideTableRow -> CollectionBatch -> TaskGroup -> FetchTask`，将定义、调度与执行连接起来。
4. 通过 RetrievalTask、NarrowIndicatorRow、ExecutionRecord 与 ScheduleJob，承接执行产物、溯源信息和运行记录。
5. 通过 `plan_version`，为范围变化、指标组变化和计划重建提供一致的版本边界。
