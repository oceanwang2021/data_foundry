## 2. 页面定位

`监控` 页面的定位不是任务明细页，也不是验收处理页，而是一个跨项目、跨需求、跨任务组的运行大盘。

与其他菜单的边界如下：

- `监控`
  - 看大盘健康度、进度率、积压和风险
  - 强调聚合状态、比例和趋势判断
  - 不展开任务实例或验收工单明细
- `采集任务管理`
  - 看任务组、采集任务、执行实例、运行细节
  - 适合定位单任务问题
- `数据验收`
  - 看验收单、审核状态、驳回与通过处理
  - 适合推进验收闭环

## 3. 设计目标

监控页需解决两个核心问题：

1. `任务运行状态健康监控`
   - 当前任务跑得稳不稳
   - 是否存在失败、取消、积压、服务异常

2. `数据状态推进监控`
   - 数据是否已经采集
   - 是否已进入审核
   - 是否已经完成闭环

设计上强调：

- 首屏即可判断系统状态
- 聚焦重点数字，不堆叠过多组件
- 用少量状态色区分健康、预警、异常
- 避免把页面做成另一个任务列表

## 4. 当前前端设计方案

### 4.1 总体视觉方向

当前页面采用简洁化设计，核心原则为：

- 白底，无浅蓝大面积渐变背景
- 去掉“运行中心驾驶舱”装饰性文案
- 减少大卡片、小卡片、圆圈图层层嵌套
- 用大数字、细进度条、轻边框做视觉层级
- 用少量分组区块突出重点

### 4.2 页面结构

页面从上到下分为 5 个区块：

#### A. 头部区

- 页面标题：`监控`
- 说明文案：统一查看系统健康、任务运行健康和数据推进进度
- 操作按钮：`刷新监控`

特点：

- 纯信息头部
- 不额外加色块背景
- 不增加装饰标签

#### B. 核心指标区

共 4 个轻量统计卡：

1. `运行健康度`
2. `任务完成率`
3. `数据采集完成率`
4. `数据审核完成率`

每个卡片仅保留：

- 指标标题
- 大号百分比数字
- 一条细进度条
- 一行说明文案

不再使用：

- 圆环图
- 右上角装饰徽标
- 卡片内复杂组件组合

#### C. 系统健康区

以简洁状态列表展示：

- 后端服务
- 调度服务
- 采集服务

每行展示：

- 服务名
- 状态标签：健康 / 预警 / 异常
- 说明文字

不再做三个并排大卡片。

#### D. 双核心监控区

左右两栏：

##### 左栏：任务运行状态监控

用于回答“任务现在跑得稳不稳”。

内容包括：

- 顶部 4 个关键数字
  - 总任务数
  - 运行中
  - 异常
  - 成功率
- 一条横向堆叠进度条展示任务状态占比
- 各状态数字拆解
- 2 到 3 条异常摘要

##### 右栏：数据状态监控

用于回答“数据推进到哪一步”。

内容包括：

- 顶部 4 个关键数字
  - 待采集
  - 待审核
  - 已完成
  - 审核通过率
- 一条阶段推进链路
  - 待采集
  - 待审核
  - 已完成
- 采集推进率与审核推进率摘要
- 2 到 3 条推进卡点提示

#### E. 风险关注区

采用轻量告警列表，而不是大色块卡片墙。

每条风险展示：

- 风险名称
- 数量
- 一句话说明
- 进入处理页面

典型风险：

- 采集异常
- 审核积压
- 系统告警

## 5. 指标口径设计

### 5.1 核心原则

监控页的统计口径统一基于已有业务实体聚合，不新造一套监控业务对象。

统一口径如下：

- `任务运行监控` 以 `fetch_tasks` 为运行状态统计源
- `数据状态监控` 以 `task_groups + acceptance_tickets` 联合判定
- `监控生命周期单元` 统一以 `task_group` 为主粒度
- 默认排除试运行数据：`triggered_by = 'trial'` 不进入正式监控口径

### 5.2 为什么以 task_group 作为数据推进粒度

原因：

- `task_group` 是一次正式采集分区/批次的天然单位
- `acceptance_tickets.task_group_id` 可直接关联
- 既能代表一次采集结果，也能进入验收闭环
- 比单个 `fetch_task` 更适合作为“推进单元”

## 6. 状态映射规则

### 6.1 任务状态监控

来源表：`fetch_tasks`

状态映射：

- `pending` -> 待执行
- `running` -> 运行中
- `completed` -> 已完成
- `failed` -> 异常
- `cancelled` -> 异常
- `invalidated` -> 异常

### 6.2 数据状态监控

联合来源：

- `task_groups.status`
- `acceptance_tickets.status`

状态判定规则：

#### 待采集

满足任一条件：

- `task_group.status = pending`
- `task_group.status = running`
- 尚未达到可审核阶段

#### 待审核

满足任一条件：

- `task_group.status in (completed, partial)` 且没有验收单
- 有验收单，但 `acceptance_ticket.status in (pending, publishing)`

#### 已完成

满足：

- `acceptance_ticket.status in (approved, partial_approved)`

#### 异常

满足任一条件：

- `task_group.status in (failed, cancelled, invalidated)`
- `acceptance_ticket.status in (rejected, fixing, publish_failed)`

## 7. 页面指标定义

### 7.1 运行健康度

这是一个综合分，不直接来自单表字段。

组成建议：

- 服务健康得分
- 任务成功率
- 数据审核推进率

当前计算思路：

- 服务健康得分占比：35%
- 任务成功率占比：40%
- 数据审核推进率占比：25%

### 7.2 任务完成率

公式：

`已完成 fetch_tasks / 全部正式 fetch_tasks`

### 7.3 数据采集完成率

公式：

`(待审核 task_groups + 已完成 task_groups) / 全部正式 task_groups`

含义：

- 已完成采集但未验收，也算“采集已完成”

### 7.4 数据审核完成率

公式：

`已完成 task_groups / 全部正式 task_groups`

### 7.5 审核通过率

公式：

`approved 或 partial_approved 的 task_groups / 已进入审核或已有验收结论的 task_groups`

## 11. 数据一致性方案

### 11.1 一致性原则

监控页不维护自己的状态副本，不新增监控业务表，所有状态都从业务真相源读取并聚合。

### 11.2 任务组聚合修正

在聚合前做一次“读修复”：

- 若 `task_group.last_aggregated_at` 为空
- 或早于该组下任一 `fetch_task.updated_at`

则先调用：

- `TaskGroupAggregateService.refreshTaskGroups(...)`

保证监控读取到的 `task_group` 状态与任务明细状态一致。

### 11.3 状态真相源约束

- 任务运行状态：以 `fetch_tasks.status` 为真相源
- 数据审核状态：以 `acceptance_tickets.status` 为真相源
- 调度状态：通过 `schedule-jobs` 下游接口读取
- 服务健康：通过 `/health` 实时探活

### 11.4 trial 数据处理

默认排除 `triggered_by = trial` 的任务组和采集任务，避免试运行扰动正式监控大盘。

## 12. 数据库设计方案

### 12.1 设计原则

本轮不新增监控业务表，不引入 `ops_snapshot`、`dashboard_summary` 之类中间表。

原因：

- 当前需求是“当前态监控”，不需要额外写快照
- 使用现有业务表即可完成聚合
- 避免双写和二次状态源带来不一致

## 17. 后续建议

若后续需要进一步增强监控能力，建议按以下方向演进：

支持从风险项一键带筛选条件跳转：

- 跳到采集任务管理并自动筛选异常任务
- 跳到数据验收并自动筛选待审核或驳回数据

## 18. 结论

当前监控页方案已经形成一条完整、可维护的数据链路：

- 前端只消费一个统一聚合接口
- 后端直接基于业务真相源聚合
- 数据口径统一以 `task_group` 为推进单元
- 任务状态、数据状态、服务健康分层明确
- 页面视觉已从“组件堆叠”收敛为“重点数字 + 条形进度 + 风险提示”

该方案既能与 `采集任务管理`、`数据验收` 清晰区分，也能保证监控页作为运行大盘的可读性和一致性。
