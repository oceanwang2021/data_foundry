# 数据生产平台 — 后端完整实现方案

## 一、现状分析

### 1.1 前端现状
- 前端基于 Next.js + TypeScript，所有数据存储在 **localStorage**（`project-requirements-storage.ts`），以 mock 数据（`mock-data.ts`、`mock-platform.ts`）作为 fallback。
- 核心业务逻辑（任务展开、计划版本管理、指标填充、稽核规则）全部在前端 `lib/` 目录中实现。
- 前端未调用任何后端 API，完全自包含运行。

### 1.2 后端现状
- FastAPI + SQLite，已有完整的数据库 Schema（8 张表）和种子数据。
- 已实现 **只读 GET 接口**（项目、需求、宽表行、任务组、采集任务、检索任务、执行记录）。
- Repository 层只有查询方法，**无 Create / Update / Delete**。
- `AgentMockScheduler` 已有基本的 mock 执行框架，但未集成到 API 路由中。

### 1.3 前后端数据模型差异

| 维度 | 前端 (types.ts) | 后端 (schemas.py) | 差异说明 |
|------|-----------------|-------------------|----------|
| Project | 含 `dataSource`（搜索引擎/知识库/固定URL） | 无 `dataSource`，仅基础字段 | 后端需扩展 |
| Requirement | 含 `processingRuleDrafts`、`requirementType` | 用 `phase` 代替 `requirementType`，含 `collection_policy` | 需对齐字段映射 |
| WideTable | 扁平结构：`schema.columns`、`dimensionRanges`、`businessDateRange` | 结构化：`WideTableSchema`（分 id/dimension/indicator/system 列）、`WideTableScope` | 后端结构更严谨，前端需适配 |
| WideTableRecord | 动态 key-value（`record[columnName]`） | 结构化 `WideTableRow`（`dimension_values`、`indicator_values`、`system_values`） | 需转换层 |
| TaskGroup | 含 `planVersion`、`groupKind`、`rowSnapshots` | 无这些字段 | 后端需扩展 |
| FetchTask | 含 `confidence`、`planVersion`、`rowBindingKey` | 含 `indicator_keys`、`dimension_values`、`execution_mode` | 互有差异，需统一 |
| ExecutionRecord | `triggeredBy` 枚举不同 | `trigger_type` 枚举不同 | 需映射 |
| BackfillRequest | 含 `taskGroupIds`、`reason` | 含 `origin`、无 `taskGroupIds` | 需扩展 |

## 五、核心业务逻辑迁移

以下业务逻辑当前在前端实现，需迁移到后端 Service 层：

### 5.1 宽表记录初始化（`wide-table-preview.ts` → `services/wide_table_service.py`）

```
输入：WideTable（含 schema、dimensionRanges、businessDateRange）
处理：
  1. 调用 business_date.expand_slots(range) 生成业务日期列表
  2. 对非业务日期的维度列做笛卡尔积
  3. 为每个 (业务日期 × 维度组合) 生成一行 WideTableRow
  4. 指标列初始化为空 IndicatorCell
  5. 系统列初始化（row_status=initialized, updated_at=null）
输出：持久化到 wide_table_rows 表
```

后端 `modeling.py` 中的 `build_rows()` 已实现此逻辑，可直接复用。

### 5.2 任务计划展开（`task-plan-reconciliation.ts` → `services/task_plan_service.py`）

```
输入：WideTable、WideTableRows
处理：
  1. 计算 plan fingerprint（基于 schema + scope + indicator groups 的哈希）
  2. 如果 fingerprint 变化，递增 plan_version
  3. 按业务日期分组记录
  4. 为每个业务日期生成 TaskGroup
  5. 为每个 TaskGroup 展开 FetchTask（row × indicator_group）
  6. 标注 groupKind（baseline / delta）
输出：持久化 task_groups + fetch_tasks
```

后端 `modeling.py` 中的 `build_task_groups()` 和 `build_fetch_tasks()` 已有基础实现，需扩展 plan version 支持。

### 5.3 任务组执行摘要（`task-group-execution.ts` → `services/execution_service.py`）

```
输入：TaskGroup、关联的 FetchTasks
处理：
  1. 统计各状态任务数（pending/running/completed/failed/skipped）
  2. 计算进度百分比
  3. 推导任务组状态（基于子任务状态组合）
输出：TaskGroupExecutionSummary
```

### 5.4 采集任务执行（`agent_mock_scheduler.py` → `services/agent_service.py`）

```
输入：FetchTask
处理：
  1. 创建 ExecutionRecord（status=running）
  2. 更新 FetchTask 状态为 running
  3. 调用 Agent Mock（模拟 1-3 秒延迟）
  4. 生成 mock 指标值
  5. 创建 RetrievalTask（含窄表行数据）
  6. 更新 WideTableRow 的 indicator_values
  7. 更新 FetchTask 状态为 completed/failed
  8. 更新 ExecutionRecord
  9. 检查并更新 TaskGroup 状态
输出：更新后的 FetchTask + WideTableRow
```

现有 `AgentMockScheduler` 已有基础框架，需补充：
- RetrievalTask 生成
- TaskGroup 状态联动更新
- 错误处理和重试逻辑

### 5.5 指标填充管线（`indicator-filling.ts` → `services/filling_service.py`）

```
输入：原始指标值、FillingRuleConfig
处理：
  1. LLM 语义解析（mock：根据值特征生成 SemanticIndicatorResult）
  2. 规则引擎依次执行 10 条填充规则：
     - FR-001: 低置信度阻塞
     - FR-002: 空值统一
     - FR-003: 日期格式转换
     - FR-004: 单位统一+指标换算
     - FR-005~007: 区间/下限/近似值处理
     - FR-008: HTML标签剔除
     - FR-009: 千分位去除
     - FR-010: 百分率转换
  3. 输出最终值 + 填充状态 + 是否需人工确认
输出：IndicatorFillingResult
```

### 5.6 数据后处理（`requirement-data-pipeline.ts` → `services/preprocessing_service.py`）

```
输入：WideTableRows、PreprocessRules
处理：
  1. 构建原始数据行（含维度值、指标值）
  2. 依次应用启用的后处理规则：
     - format_fix: 格式修复（数值标准化）
     - null_fix: 空值语义修复
     - unit_convert: 单位换算
     - derived: 衍生指标计算
  3. 输出后处理结果行
输出：处理后的数据行
```

### 5.7 数据稽核（`requirement-data-pipeline.ts` → `services/audit_service.py`）

```
输入：后处理结果行、AuditRules
处理：
  1. 对每行数据逐条评估稽核规则
  2. 区分阻断式（blocking）和非阻断式（non_blocking）
  3. 阻断式未通过 → 标记异常，阻止入库
  4. 非阻断式未通过 → 标记警告，允许入库
输出：AuditResultRow[]
```

### 5.8 补采流程（`BackfillRequest` → `services/backfill_service.py`）

```
输入：需求ID、宽表ID、业务日期范围、原因
处理：
  1. 创建 BackfillRequest
  2. 为指定日期范围生成 TaskGroup（source=backfill）
  3. 展开 FetchTask
  4. 触发执行
输出：BackfillRequest + 关联的 TaskGroups
```

后端 `modeling.py` 中的 `build_backfill_requests()` 已有基础实现。

## 十、关键设计决策

1. **数据库选型**：继续使用 SQLite，满足单机部署需求。如需多实例部署，可后续迁移到 PostgreSQL（Schema 兼容）。

2. **异步执行**：Agent Mock 执行使用 `asyncio` + 后台任务（`BackgroundTasks`），避免阻塞 API 响应。批量执行使用 `asyncio.gather` 并发。

3. **事务管理**：所有写操作使用 SQLite 事务（`connection.commit()`），确保数据一致性。任务执行涉及多表更新时，使用显式事务。

4. **前后端数据映射**：在 API 响应层做字段映射（snake_case ↔ camelCase），使用 Pydantic 的 `alias_generator` 或自定义序列化。

5. **Agent 接口抽象**：定义 `AgentService` 抽象基类，Mock 实现继承该基类。未来替换真实 Agent 时只需实现新的子类。

6. **幂等性**：任务执行接口设计为幂等（重复调用不会创建重复记录），通过任务状态检查实现。

7. **向后兼容**：前端改造期间保留 localStorage fallback，通过环境变量 `NEXT_PUBLIC_USE_API=true` 控制切换。
