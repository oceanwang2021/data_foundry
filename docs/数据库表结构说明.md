# 数据库表结构说明

> 本文档基于 `DDD-数据库表改造方案.md` 和现有 Flyway 迁移脚本整理，描述项目中所有表的功能和承载信息。

---

## 一、数据库架构总览

| 数据库 | 所属服务 | 表数量 | 用途 |
|--------|---------|--------|------|
| `data_foundry_backend` | backend-service | 14张 | 项目、需求、宽表定义、任务组、子任务、采集记录等元数据 |
| `data_foundry_scheduler` | scheduler-service | 1张 | 调度作业记录 |
| `target_tables` | 通用（共享读） | 动态增长 | 最终产出的目标物理表（动态创建） |

## 四、Target Tables 数据库：动态目标表

### 数据库用途

`target_tables` 是一个**共享读**的数据库，用于存储最终产出的目标物理表。这些表是**动态创建**的，不需要在数据库初始化时预先定义。

### 核心设计原则

1. **动态创建**：根据 `wide_tables` 表中的 `schema_json` 定义，运行时动态创建目标表
2. **共享读**：backend-service 有 SELECT 权限，scheduler-service 有 SELECT 权限，用于数据查询和展示
3. **命名规范**：表名存储在 `wide_tables.table_name` 中（如 `dwd_sales_daily`）

### 目标表结构规范

#### 必需字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `(业务主键)` | 动态 | 由 `schema_json` 中 `is_pk: true` 的字段构成复合主键 |
| `_batch_id` | VARCHAR(64) | 批次标识（用于数据对比） |
| `_task_id` | VARCHAR(64) | 任务ID（追溯执行来源） |
| `_fetch_task_id` | VARCHAR(128) | 采集子任务ID |
| `_collected_at` | DATETIME | 采集时间 |

#### 可选字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `(业务字段)` | 动态 | 由 `schema_json` 定义的维度和指标字段 |
| `_created_at` | DATETIME | 记录创建时间（默认 CURRENT_TIMESTAMP） |

#### 典型建表示例

```sql
CREATE TABLE dwd_sales_daily (
  -- 业务主键（维度）
  stat_date VARCHAR(32) NOT NULL,
  region_code VARCHAR(32) NOT NULL,
  product_code VARCHAR(32) NOT NULL,
  
  -- 业务指标
  sales_amount DECIMAL(18,2),
  order_count INT,
  customer_count INT,
  
  -- 元数据字段
  _batch_id VARCHAR(64) NOT NULL,
  _task_id VARCHAR(64),
  _fetch_task_id VARCHAR(128),
  _collected_at DATETIME,
  
  PRIMARY KEY (stat_date, region_code, product_code, _batch_id),
  INDEX idx_dwd_sales_daily_biz_date (stat_date),
  INDEX idx_dwd_sales_daily_batch (_batch_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
  COMMENT='dwd 月度销售宽表';
```

### 典型数据写入逻辑

```sql
-- UPSERT 语句（存在则更新，不存在则插入）
INSERT INTO dwd_sales_daily 
  (stat_date, region_code, product_code, sales_amount, _batch_id, _task_id, _collected_at)
VALUES 
  ('2024-12-01', 'SH', 'A001', 1000.00, 'batch_20241201', 'task_001', NOW())
ON DUPLICATE KEY UPDATE
  sales_amount = VALUES(sales_amount),
  _task_id = VALUES(_task_id),
  _collected_at = VALUES(_collected_at);
```

### 数据对比机制

数据对比基于 **`_batch_id`** 实现：

```
上期批次: batch_20241130          本期批次: batch_20241201
┌─────────────────────┐          ┌─────────────────────┐
│ biz_date │ amount   │          │ biz_date │ amount   │
├─────────────────────┤          ├─────────────────────┤
│ 20241201 │ 1000    │          │ 20241201 │ 1100    │ ← 变化
│ 20241202 │ 2000    │          │ 20241202 │ 2000    │ ← 相同
│ 20241203 │ 1500    │          │ 20241203 │ 3000    │ ← 新增
└─────────────────────┘          └─────────────────────┘
```

## 七、数据采集完整流程

```
1. 需求定义
   └─ wide_tables.schema_json 定义目标表结构
   
2. 目标表创建（首次执行时）
   └─ TargetTableManager 根据 schema_json 动态建表
   
3. 任务生成
   └─ task_groups 划分采集任务批次
   └─ fetch_tasks 生成子任务
   
4. 数据采集
   └─ 执行器执行 fetch_tasks
   
5. 数据写入
   └─ UPSERT 到 target_tables.¡dwd_sales_daily¡
   
6. 历史对比
   └─ 基于 _batch_id 字段对比不同批次数据
```

## 八、总结

| 维度 | 现状 | 说明 |
|------|------|------|
| **DB 数量** | 3个 | Backend 14张 + Scheduler 1张 + Target Tables 动态 |
| **表数量** | Backend 固定 14张 | 加上目标表（动态增长） |
| **DDD 分层** | 清晰：Project/Requirement/Task/Backfill/Audit | 表 ownership 明确 |
| **数据完整性** | 全量索引已补齐（V002） | 覆盖所有列表/过滤场景 |
| **幂等设计** | FetchTask/TaskGroup 按 PK 幂等 | Target Table 按复合主键幂等 |
| **扩展预留** | execution_records 表（M3b阶段） | 用于执行侧追踪 |
| **动态表管理** | target_tables 数据库 | 通过 API 查询表和列信息 |
