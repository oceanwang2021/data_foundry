二、按优先级拆分
第 1 步：先拆“纯函数”和类型，风险最低

你文件底部有很多纯工具函数，例如：

buildTaskPlanView
buildTaskGroupRunViews
buildTaskGroupRunSections
buildTaskInstanceRowViews
formatTaskInstanceParameterLines
buildTrialParameterRowKey
summarizeDimensions
calculateDimensionCombinationCount
taskFrequencyLabel
getTriggerDisplayLabel
getTaskGroupStatusLabel
materializeLocalTaskGroupArtifacts
buildLocalFetchTasks

这些基本不依赖 React state，适合先移动到：

utils/requirementTaskPlan.ts
utils/requirementTaskViews.ts
utils/requirementTaskFormatters.ts
utils/requirementTaskLocalExecution.ts
types.ts

这是最适合先让 Codex 做的，因为不改变页面结构，出错概率低。

第 2 步：拆三个大 Tab

当前页面实际已经有三个子模块：

prompts：采集提示词管理
tasks：采集任务
output：数据产出

建议拆成：

PromptManagementTab.tsx
TaskExecutionTab.tsx
TaskOutputTab.tsx

RequirementTasksPanel.tsx 中只保留：

{activeTaskSubTab === "prompts" && (
  <PromptManagementTab ... />
)}

{activeTaskSubTab === "tasks" && (
  <TaskExecutionTab ... />
)}

{activeTaskSubTab === "output" && (
  <TaskOutputTab ... />
)}

这样主文件会立刻从 4000+ 行降到 1500～2000 行左右。

第 3 步：拆任务实例表格

你现在有明显的 renderTaskInstanceTable、renderSelectableTaskInstanceTable、renderTaskInstanceActions、renderSelectableTaskInstanceActions 这类 render helper。

这些应该拆成组件：

TaskInstanceTable.tsx
TaskInstanceActions.tsx
TaskStatusLegend.tsx

尤其是任务实例表格，后面很可能继续改“采集”“重采”“取消”“查看日志”“批量采集”，如果继续放在主页面里，会越来越难维护。

第 4 步：拆批量执行逻辑为 Hook

这部分逻辑风险较高，但也最值得拆：

batchExecutionStateByTaskId
batchExecutionTaskOrderByScope
bulkExecutingScopeKeys
selectedTaskIdsByScope
runningTaskIds
cancellingTaskIds
handleBatchExecuteTasks
dispatchSingleTaskExecution
handleCancelTask

建议抽成：

hooks/useTaskExecutionQueue.ts

返回：

{
  runningTaskIds,
  cancellingTaskIds,
  bulkExecutingScopeKeys,
  selectedTaskIdsByScope,
  getTaskInstanceDisplayStatus,
  getTaskInstanceDisplayCollectionTaskId,
  canSelectTaskInstance,
  handleToggleTaskSelection,
  handleToggleAllTaskSelection,
  handleBatchExecuteTasks,
  handleRequestTaskRerun,
  handleCancelTask,
}

这个不要第一步做，因为它依赖很多 props 和回调，适合在 UI 组件先拆完之后再做。

第 5 步：拆提示词和指标分组逻辑

这一块可以拆成两个 Hook：

对应当前逻辑：
