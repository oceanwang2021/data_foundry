# 采集任务实例状态与采集接口返参对应

## 1. 说明

本文梳理页面中采集任务实例的 5 个状态：

- `pending`：待执行
- `running`：执行中
- `completed`：已完成
- `failed`：失败
- `cancelled`：已取消

以及它们和 4 类采集接口之间的对应关系：

1. 生成任务：`POST /api/search`
2. 查询状态：`GET /api/task/{task_id}/status`
3. 查询结果：`GET /api/task/{task_id}/result`
4. 取消任务：`POST /api/task/{task_id}/cancel`

文中同时区分两层接口：

- 下游采集接口：由 `CollectionSearchClient` 调用
- 本系统 facade 接口：前端实际调用的接口

页面里“实例状态”本质上对应本地 `fetch_task.status`，它是一个“本地聚合状态”，不等价于下游任务状态原样透传。

## 2. 关键接口与返回结构

### 2.1 下游采集接口在后端内部的对象

#### 生成任务 `createSearch`

```json
{
  "success": true,
  "taskId": "downstream-task-id",
  "errorMessage": null
}
```

失败时：

```json
{
  "success": false,
  "taskId": null,
  "errorMessage": "missing task_id / success=false / http 5xx ..."
}
```

#### 查询状态 `getTaskStatus`

```json
{
  "success": true,
  "taskId": "downstream-task-id",
  "status": "pending|running|completed|failed|cancelled",
  "errorMessage": null,
  "rawResponseJson": "{...下游原始响应...}"
}
```

失败时：

```json
{
  "success": false,
  "taskId": "downstream-task-id",
  "status": null,
  "errorMessage": "missing task status / success=false / http 5xx ...",
  "rawResponseJson": "{...如果有原始响应...}"
}
```

#### 查询结果 `getTaskResult`

```json
{
  "success": true,
  "taskId": "downstream-task-id",
  "status": "completed",
  "finalReport": "...",
  "rawResponseJson": "{...下游原始响应...}",
  "errorMessage": null
}
```

失败时：

```json
{
  "success": false,
  "taskId": "downstream-task-id",
  "status": null,
  "finalReport": null,
  "rawResponseJson": "{...如果有原始响应...}",
  "errorMessage": "success=false / non-2xx / unavailable ..."
}
```

#### 取消任务 `cancelTask`

```json
{
  "success": true,
  "taskId": "downstream-task-id",
  "message": "cancelled",
  "errorMessage": null,
  "httpStatusCode": 200
}
```

失败时：

```json
{
  "success": false,
  "taskId": "downstream-task-id",
  "message": null,
  "errorMessage": "detail / message / error / http 409 ...",
  "httpStatusCode": 409
}
```

### 2.2 前端实际调用的本系统接口

#### 执行单个实例

`POST /api/tasks/{taskId}/actions/execute`

成功返回：

```json
{
  "ok": true,
  "task_id": "fetch-task-id",
  "collection_task_id": "downstream-task-id",
  "status": "running|failed"
}
```

说明：

- 发起成功后通常返回 `status=running`
- 如果在“生成下游任务”这一步就失败，会直接返回 `status=failed`

#### 重采单个实例

`POST /api/tasks/{taskId}/retry`

成功返回：

```json
{
  "ok": true,
  "task_id": "fetch-task-id",
  "collection_task_id": "downstream-task-id",
  "status": "running|failed"
}
```

#### 查询状态明细

`GET /api/tasks/{taskId}/status-detail`

优先透传下游原始响应；如果无法直接透传，则返回：

```json
{
  "success": true,
  "data": {
    "task_id": "downstream-task-id",
    "status": "running|completed|failed|cancelled",
    "fetch_task_id": "fetch-task-id",
    "collection_task_id": "downstream-task-id"
  },
  "detail": "仅失败时可能出现"
}
```

#### 查询本地结果

`GET /api/tasks/{taskId}/results`

返回的是本地落库结果，不是下游 `/result` 原始响应：

```json
{
  "collectionResults": [
    {
      "id": "collection-result-id",
      "fetchTaskId": "fetch-task-id",
      "externalTaskId": "downstream-task-id",
      "status": "completed|success|partial|not_found|conflict|failed",
      "finalReport": "...",
      "rawResultJson": "{...}",
      "errorMsg": "..."
    }
  ],
  "collectionResultRows": [
    {
      "id": "row-id",
      "fetchTaskId": "fetch-task-id",
      "indicatorKey": "...",
      "status": "accepted|rejected|not_found",
      "warningMsg": "...",
      "whyNotFound": "..."
    }
  ]
}
```

#### 取消实例

`POST /api/tasks/{collectionTaskId}/actions/cancel`

成功返回：

```json
{
  "ok": true,
  "task_id": "fetch-task-id",
  "collection_task_id": "downstream-task-id",
  "status": "cancelled",
  "message": "任务已取消"
}
```

失败时返回非 2xx，响应体是统一异常包装：

```json
{
  "code": 409,
  "message": "Task is not running",
  "data": null
}
```

或：

```json
{
  "code": 409,
  "message": "下游返回的 detail/message/error",
  "data": null
}
```

前端 `cancelTask()` 本身不返回对象；如果取消失败，会直接抛出 `Error(message)`，页面最终展示：

```text
取消失败：<message>
```

## 3. “取消采集任务时报失败”时，返参到底是什么

这个问题需要分 3 层看：

### 3.1 下游采集取消接口的内部返参

后端调用下游 `POST /api/task/{task_id}/cancel` 后，会先被解析成：

```json
{
  "success": false,
  "taskId": "downstream-task-id",
  "message": null,
  "errorMessage": "detail / message / error / http 409 ...",
  "httpStatusCode": 409
}
```

也就是说，取消失败时最关键的字段是：

- `success=false`
- `taskId=下游任务ID`
- `errorMessage=失败原因`
- `httpStatusCode=下游 HTTP 状态码`

### 3.2 本系统取消接口对前端的 HTTP 返参

本系统收到上面的失败结果后，不会返回 `ok=false` 的业务对象，而是直接抛 `ResponseStatusException`。

因此前端实际收到的是“非 2xx + 统一异常包装”：

```json
{
  "code": 409,
  "message": "Task is not running / 下游错误明细 / Failed to cancel collection task",
  "data": null
}
```

常见情况：

| 场景 | HTTP 状态 | 响应体 |
| --- | --- | --- |
| 本地实例不是 `running` | `409` | `{"code":409,"message":"Task is not running","data":null}` |
| 本地实例没有 `collection_task_id` | `409` | `{"code":409,"message":"Task has no collection task id","data":null}` |
| 下游返回 409/404/500 等 | 与下游状态码一致 | `{"code":下游状态码,"message":"下游 detail/message/error","data":null}` |
| 下游不可用 | `503` | `{"code":503,"message":"unavailable: ...","data":null}` |

### 3.3 前端最终拿到的内容

前端 `apiFetch()` 会优先从错误响应体里提取：

1. `detail`
2. `message`
3. `error`

而本系统统一异常包装里是 `message`，所以前端最终拿到的是：

```text
Error("Task is not running")
```

页面文案最终表现为：

```text
取消失败：Task is not running
```

## 4. 5 个实例状态与 4 类采集接口返参的对应关系

> 注意：这里的“对应”是“该状态下最常见、最有代表性的返参特征”，不是说每个状态都一定会调用 4 个接口。

| 实例状态 | 本地含义 | 生成任务 `POST /api/search` | 查询状态 `GET /status` | 查询结果 `GET /result` / 本地 `/results` | 取消任务 `POST /cancel` |
| --- | --- | --- | --- | --- | --- |
| `pending` | 本地已建实例，但尚未向下游成功发起任务 | 通常尚未调用；如果刚重置待重采，也可能尚未再次发起 | 通常无法查，因为往往没有 `collection_task_id`；调本地 `/status-detail` 常见返回 `409 Task has no collection task id` | 本地 `/results` 通常为空数组；下游 `/result` 通常无意义 | 本地取消接口直接返回 `409 Task is not running` |
| `running` | 已成功生成下游任务，正在执行 | `success=true`，且返回 `task_id`；本地执行接口返回 `ok=true`、`collection_task_id`、`status=running` | 通常 `success=true`，`data.status=running` | 本地 `/results` 通常还没有结果，或者结果未落库；下游 `/result` 常见为未完成/不可取 | 如果下游取消成功，本地取消接口返回 `ok=true`、`status=cancelled`；如果失败，返回非 2xx 异常包装 |
| `completed` | 本地认为任务完成，且可用结果已落库 | 早先一定经历过一次生成成功，即曾拿到 `task_id` | 通常 `success=true`，`data.status=completed` | 下游 `/result` 通常 `success=true`、包含 `finalReport`；本地 `/results` 通常有 `collectionResults` 和 `collectionResultRows` | 本地取消接口不会再调用下游，直接返回 `409 Task is not running` |
| `failed` | 本地认为任务失败 | 有两类：1）生成时就失败，此时 `success=false`、没有 `task_id`；2）生成成功后执行阶段失败，此前曾成功拿过 `task_id` | 也分两类：1）生成即失败，通常无法查状态；2）执行阶段失败，常见 `success=true` 且 `data.status=failed` | 也分两类：1）生成即失败，本地 `/results` 为空；2）执行阶段失败，本地 `/results` 可能已有 `collectionResults`，其中 `status=failed`，也可能带 `errorMsg` | 本地取消接口直接返回 `409 Task is not running`；如果实例虽然本地是 `failed`，但误传下游任务号来取消，也不会通过本地前置校验 |
| `cancelled` | 本地下游任务已成功取消 | 早先一定经历过一次生成成功，即曾拿到 `task_id` | 后续查状态时，常见 `success=true`，`data.status=cancelled` | 通常没有可用结果；本地 `/results` 常为空 | 再次取消时，本地取消接口直接返回 `409 Task is not running` |

## 5. `failed` 状态为什么最容易和接口返参“看起来不一致”

`failed` 不是单一来源，它至少有 3 类来源：

### 5.1 生成下游任务时失败

典型特征：

- `createSearch.success=false`
- 没有 `taskId`
- 本地实例直接写成 `failed`
- 后续大概率没有 `collection_task_id`

此时：

- 查状态通常没法查
- 查结果通常为空
- 取消一定会报 `Task is not running`

### 5.2 下游执行状态就是失败

典型特征：

- 生成任务成功，已有 `collection_task_id`
- `getTaskStatus.success=true`
- `getTaskStatus.status=failed`
- 本地同步后实例状态变成 `failed`

此时：

- 状态接口与本地状态一致
- 结果接口可能为空，也可能有失败结果

### 5.3 下游执行结束了，但结果处理后仍被本地判失败

典型特征：

- 下游任务可能已经 `completed`
- 但 `agentResult` 缺失，或 `agentResult.status=failed`
- 或 `normalizedRows` 为空，本地也会把实例判成 `failed`

此时会出现一个容易误解的现象：

- 本地实例状态：`failed`
- 下游状态查询：可能仍然显示 `completed`

这是因为页面实例状态看的是本地 `fetch_task.status`，它包含了“结果可用性校验”这一步，而不是只看下游任务跑没跑完。

## 6. 联调时建议优先看的字段

### 6.1 看实例是否成功生成下游任务

优先看：

- `fetch_task.collection_task_id`
- 执行接口返回里的 `collection_task_id`

如果没有这个字段，说明大概率还没真正生成下游任务。

### 6.2 看实例状态为何失败

优先按顺序看：

1. 执行接口返回的 `status`
2. `/api/tasks/{taskId}/status-detail`
3. `/api/tasks/{taskId}/results`
4. `collectionResults[].status`
5. `collectionResults[].errorMsg`
6. `collectionResultRows[].status / warningMsg / whyNotFound`

### 6.3 看取消失败到底卡在哪一层

优先按顺序看：

1. 本地实例当前 `status` 是否为 `running`
2. `collection_task_id` 是否存在
3. 取消接口 HTTP 状态码
4. 取消接口响应体里的 `message`

最常见并且最正常的失败其实是：

```text
409 Task is not running
```

这通常不是下游异常，而是本地前置校验拦住了。
